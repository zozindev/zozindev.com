브라우저 전용 도구라는 말은 듣기에는 단순하지만, 실제로는 확인할 부분이 있다. 웹사이트에서 열린다고 해서 전부 브라우저 안에서 처리되는 것은 아니다. 파일을 고르는 순간 서버로 업로드할 수도 있고, 입력한 텍스트를 저장할 수도 있고, 결과를 만들기 위해 외부 API로 보낼 수도 있다. 그래서 민감한 입력값을 넣는 도구라면 처리 위치를 먼저 봐야 한다.

첫 번째 확인점은 업로드 버튼의 의미다. 이미지 편집 도구에서 “업로드”라고 쓰여 있어도, 파일을 서버에 보내는 업로드인지 브라우저에서 읽는 파일 선택인지 다를 수 있다. 설명에 “서버로 전송하지 않음”, “브라우저에서 처리”, “로컬에서만 렌더링” 같은 안내가 있으면 판단하기 쉽다. 안내가 전혀 없고 계정 로그인을 요구한다면 더 조심해서 봐야 한다.

두 번째는 네트워크가 끊겨도 기본 기능이 되는지다. 완전히 오프라인에서 돌아갈 필요는 없지만, 이미 열린 페이지에서 파일 선택과 편집이 계속 된다면 브라우저 처리 가능성이 높다. 반대로 버튼을 누를 때마다 서버 응답을 기다리거나, 결과를 만들 때 업로드 진행률이 뜬다면 서버 처리일 가능성이 있다.

세 번째는 저장 위치다. 브라우저 전용 도구는 보통 결과를 바로 다운로드하거나, 브라우저 저장소에만 임시로 둔다. 계정별 작업함, 클라우드 보관함, 공유 링크 생성 기능이 있다면 서버 저장이 들어갔을 수 있다. 이것이 무조건 나쁘다는 뜻은 아니다. 다만 캡처 이미지나 내부 문구처럼 민감한 입력값을 다룰 때는 사용자가 알고 선택해야 한다.

네 번째는 개인정보처리방침과 가이드 페이지다. 작은 개인 도구라도 어떤 데이터를 받는지, 서버 저장을 하는지, 외부 서비스로 이동하는지 정도는 설명되어 있어야 한다. 설명이 길 필요는 없지만 기준은 있어야 한다. <a href="/projects/zozincanvas/">zozincanvas</a>와 <a href="/projects/text-to-img/">TEXT_TO_IMG</a>를 정적 도구로 둔 이유도 입력값을 굳이 서버로 보내지 않기 위해서다.

다섯 번째는 결과물의 소유와 재사용이다. 텍스트 이미지 생성기라면 입력한 문구가 서비스에 저장되는지, 썸네일 예시로 재사용되는지 확인해야 한다. 정산 도구라면 멤버 이름과 지출 내역이 서버에 남는지 봐야 한다. 저장이 필요 없는 도구라면 애초에 저장하지 않는 구조가 더 단순하고 안전하다.

여섯 번째는 기능과 책임의 균형이다. 서버 저장, 로그인, 동기화, 공유 링크, 협업 편집은 편하다. 대신 운영자는 데이터를 관리해야 하고 사용자는 데이터가 어디에 있는지 신경 써야 한다. 반대로 브라우저 전용 도구는 기기를 바꾸면 이어서 작업하기 어렵고, 브라우저 데이터를 지우면 설정이 사라질 수 있다. 둘 중 무엇이 맞는지는 도구의 목적에 따라 다르다.

일곱 번째는 입력값의 민감도다. 공개해도 되는 짧은 문구라면 온라인 도구를 써도 큰 부담이 없을 수 있다. 하지만 API 키, 고객 이름, 결제 화면, 계좌 정보, 내부 URL이 들어간 이미지는 다르다. 이런 입력값은 서버 업로드 없는 도구가 더 잘 맞는다. 편의성보다 노출 가능성을 줄이는 쪽이 먼저다.

브라우저 전용 도구를 고를 때는 질문을 단순하게 잡으면 된다. 파일이나 문구가 서버로 가는가. 저장되는가. 계정이 필요한가. 결과를 만들기 위해 외부 API를 부르는가. 개인정보 처리 기준이 설명되어 있는가. 이 질문에 답이 안 보이면 민감한 입력값은 넣지 않는 편이 낫다.

개인 웹 도구는 작을수록 기준이 명확해야 오래 쓸 수 있다. 모든 기능을 넣는 것보다 어떤 데이터도 굳이 받지 않는 편이 나을 때가 많다. 브라우저 안에서 끝나는 도구는 기능이 적어 보일 수 있지만, 이미지 가리기, 텍스트 이미지 만들기, 단순 정산처럼 짧게 끝나는 작업에는 오히려 그 단순함이 장점이다.
